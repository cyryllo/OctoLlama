# OctoLlama — user guide

This document explains **how OctoLlama is built** (one master, many AI hosts)
and **how to use it day to day**. It assumes the `install.sh` installation has
already been done (see README).

---

## 1. How it works: one master, many hosts

Several computers on your home network can run Ollama — e.g. your main
computer, a mini-PC under the TV, and an old laptop in a closet. OctoLlama
ties them together into one whole:

- **Master** — the computer running the OctoLlama web panel (port 5000).
  This is the only place you open in a browser. The master also runs its own
  Ollama, the LiteLLM aggregator (port 4000), and Open WebUI (chat).
- **Slave** — any additional computer running Ollama. It has no panel of its
  own. It only runs Ollama and a small daemon that carries out commands from
  the master.

**The key rule:** you control everything from a single panel on the master.
You never need to log into a slave over SSH to start a service, pull a model,
or change a setting — you do it all from a browser, including from your phone.

### What the end user sees (chat, VS Code)

Models from **all** hosts are available at **one address**: LiteLLM on the
master (port 4000). Thanks to that:

- **Open WebUI** (chat) sees a single list of models — it doesn't care which
  computer physically computes the answer.
- **Continue.dev in VS Code** gets a single config generated by the panel,
  and it too draws on every host at once.
- In the panel (the **LLM** tab) you decide yourself **which models from
  which hosts** are exposed at all.

### How the master talks to slaves (for the curious)

The web panel never runs commands as root. Instead, it writes "what the user
wants" to a `state.json` file. On every host (the master and each slave) a
local daemon with root privileges watches that file, applies changes
(start/stop the service, environment variables) and writes the result back to
`status.json`.

State files reach slaves over **NFS**: the master exports a separate
directory per host (restricted to that host's IP address), and the slave
mounts it. The root daemon never opens any network port — its only contact
with the outside world is a file on disk. That's a deliberate security
decision.

## 2. The panel's four tabs

The panel is available at `http://<master-address>:<port>` (after logging
in) — you set the port once during installation (default 5000, Enter accepts
the default).

### Master
Controls Ollama on the management host:

- start / stop / autostart the service,
- environment variables affecting performance (context size, VRAM,
  Vulkan/iGPU, KV cache, network availability),
- a link to model management (list, sizes, what's currently loaded in
  memory, pulling with a progress bar, deleting).

### Slave
Adding and removing remote hosts:

1. Click **Add host**, enter a name and IP address.
2. The panel generates a ready-made `install-<name>.sh` script.
3. Download it and run it **once, by hand** on the new machine (e.g. over
   SSH). The script installs Ollama, mounts the state directory over NFS
   (retrying for a bit if the export on the master isn't ready yet), and
   sets up its daemon.
4. From then on the host shows up in the panel with its own icon-button row
   (hover each for a label): its own "Ollama settings" page — the exact same
   start/stop/autostart/environment-variable controls as the Master tab,
   just for that host — plus model management, re-downloading the
   installer, and removing it.

This is also where **power management** lives: Wake-on-LAN (wake up a
sleeping or powered-off host — MAC address shown under its IP, auto-detected
from ARP or entered manually) and remote power off / restart / suspend —
each asks for confirmation before acting. Typical scenario: a mini-PC sleeps
and draws no power, you wake it from your phone when you need extra
compute, and suspend it again from the same tab once you're done.

### LLM
Controls the LiteLLM aggregator:

- start / stop the aggregator,
- choose **which models from which hosts** get exposed under the shared
  address (port 4000, OpenAI-compatible API),
- generate a **Continue.dev** config for VS Code from the currently exposed
  models — for manual pasting (the panel never overwrites your file),
- **load balancing and reliability** (see below) — works out of the box as
  soon as the same model is checked on more than one host; everything else
  is optional.

#### Load balancing and reliability

If the same model (exactly the same name and tag, e.g. `qwen2.5-coder:14b`)
is exposed on **more than one host**, the panel immediately shows it as
"balanced across N hosts" — LiteLLM automatically spreads requests between
them, nothing more to click. Below that, in the same tab, are optional
settings to fine-tune this behavior:

- **Routing strategy** — how LiteLLM picks a host among the balanced ones:
  - `simple-shuffle` — random (default, simplest),
  - `least-busy` — the host currently handling the fewest in-flight requests,
  - `latency-based-routing` — the host that answered fastest most recently,
  - `usage-based-routing-v2` — takes usage limits (tokens/requests per
    minute) into account.
- **Host priority** — for a model on several hosts you can set an order
  (lower number = higher priority, e.g. your faster machine as "1"). Leave
  it blank to keep pure random/automatic balancing.
- **Fallbacks** — for each exposed model you can pick a backup model from
  the list. When a request to the primary model fails, LiteLLM automatically
  tries the backup (e.g. a small 7b model as a backup for a bigger 14b one,
  should it fail).
- **Retry / timeout / cooldown** — how many times to retry a failed request,
  after how many seconds to consider it dead, and how long to "cool down"
  (temporarily skip) a host after a run of errors.
- **Context window fallback** (a separate checkbox) — when a request is too
  long for a small model's context window, enabling this makes LiteLLM
  automatically try the larger model you picked instead of just returning
  an error.

**Important:** saving these settings (or changing which models are exposed
above) **immediately restarts the running LiteLLM service** with the new
config, so changes take effect right away — no need to click Start/Stop
separately. If you have
something added by hand in
`~/.config/octollama/litellm_config.yaml` (e.g. another provider's
model with its own API key), the panel won't overwrite it — it merges its
own entries with yours every time the service starts.

#### Model roles for Continue.dev

Different models are good at different jobs — a small model for quick
autocomplete, a bigger one for chat/agent work with tools. In the same LLM
tab, each exposed model only gets checkboxes for the Continue.dev roles its
capabilities actually support: `chat`, `edit`, `apply`, `autocomplete`,
`embed`, `rerank`. This is based on the capabilities Ollama reports for that
model right next to it (e.g. `tools, insert, completion`) — `tools` support
is required for `edit`/`apply`, `insert` for `autocomplete`, `embedding` for
`embed`/`rerank`, and `completion` for `chat`; a model without the right
capability just doesn't get that checkbox at all, so you can't assign it a
role that would fail in Continue anyway (a model with no reported
capabilities at all isn't restricted, since there's nothing to check
against). This only changes the generated Continue config below — it
doesn't touch the running LiteLLM service, so saving it doesn't restart
anything.

### WebUI
Start / stop **Open WebUI** — the browser chat. It's wired to LiteLLM, so it
sees exactly the models you picked in the LLM tab — from every host at once.

## 3. Common scenarios

**I want to chat with a model from my phone.**
Open the panel → WebUI tab → start Open WebUI → open its address in your
phone's browser. Done — the chat runs on models from your whole network.

**I want to add a second computer with Ollama.**
Slave tab → Add host → run the generated installer on the new machine.

**Model X needs to be available in VS Code.**
Master tab → model management → pull the model (on the host you choose).
Then LLM tab → check the model to expose it → generate the Continue.dev
config → paste it on your machine.

**I have the same model on two computers and want it to actually matter.**
LLM tab → check that model on both hosts (checkboxes in the "Exposed
models" section) → save selection. The panel immediately shows
"balanced across 2 hosts" — LiteLLM spreads requests on its own. Optionally
you can also set a priority (e.g. your faster computer as "1") or a
fallback to a smaller model, in case both hosts happen to fail.

**A host stopped responding.**
Check its status in the Master tab. If it's asleep or powered off — Slave
tab → Wake-on-LAN. If it's running but hung — remote restart from the same
tab.

**I changed a setting, but nothing happens.**
Changes are applied by the daemon (`octollama-daemon`). If it's
stopped, commands from the panel wait in `state.json` until it starts.
Check: `sudo systemctl status octollama-daemon` (on the host the
change concerns).

## 4. Security — what's worth knowing

- The panel requires login (single user, password hash in a local file, no
  database).
- The root daemon doesn't listen on any port — communication is file-based
  only.
- NFS exports are restricted per host IP address.
- The panel has no TLS — if you want HTTPS, put a reverse proxy in front of
  it (e.g. Nginx Proxy Manager). That's a deliberate design decision.
- OctoLlama is meant for a **home / trusted network** — don't expose port
  5000 or 4000 directly to the internet.
